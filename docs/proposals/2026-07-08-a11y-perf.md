# Proposal: Accessibility & Performance Budgets

- Status: Draft (design proposal) · Date: 2026-07-08 · Relates to: warden-next-competitive-gaps.md §1.7

## Summary

This proposal adds two new CTRF-producing checks to Warden's existing tier + gate pipeline: an
**accessibility check** (axe-core against the routes a PR actually changed, surfaced as findings
with severity) and a **performance-budget check** (Lighthouse metrics for those same routes,
compared against configured budgets). Both reuse the exact converter → gate → CTRF shape Warden
already ships for k6 and OWASP ZAP, so they plug into the merge gate and the dashboard without any
new reporting surface.

## Motivation

Gap analysis §1.7 notes that axe-core-based accessibility scanning is table stakes across the
industry, and that Lighthouse CI / performance budgets are a standard release gate wherever
frontend teams ship UI — accessibility is increasingly a compliance requirement, and performance
budgets catch regressions that functional tests never touch (a page that still "passes" every
assertion but got 400ms slower or lost a color-contrast ratio). Today Warden's tier system decides
_which functional tests_ run against a diff; it has no equivalent notion of "does this diff's UI
still meet its accessibility and performance bar." §1.7 explicitly scopes the fix as two checks
that "slot into the existing tier + gate model" — this proposal is that design, grounded in the
converter/gate pattern Warden already uses for `runner/src/perf/k6.ts` (load-test thresholds) and
`runner/src/security/zap.ts` (OWASP alerts).

## Goals

1. On a PR, resolve the **routes** a diff actually touches (not the whole app) from the
   orchestrator's `ChangeSurface`, using a declared route map.
2. Run axe-core against each resolved route and turn violations into CTRF tests with severity,
   WCAG tags, and enough detail (selector, help URL) for a reviewer to act without re-running
   anything locally.
3. Run Lighthouse against each resolved route and turn configured budget breaches (performance
   score, LCP, TBT, CLS) into CTRF tests the same way.
4. Feed both into the same merge gate as every other tier: BLOCK on the severities/breaches a team
   configures as blocking, WARN on the rest, PASS otherwise — combined with the existing
   functional-test gate into one decision.
5. Reuse `@warden/runner`'s existing pattern (pure converter + pure gate + one un-unit-tested
   integration function that shells out) so both checks are hermetically testable.

## Non-Goals

- Full-site accessibility/performance audits on every push — this is diff-scoped, matching how
  Warden already scopes functional tiers to the change surface.
- Visual regression or Lighthouse's non-performance categories (SEO, best-practices) — performance
  and accessibility only, matching §1.7's scope. Best-practices/SEO budgets are a natural follow-on
  using the same shape.
- A bespoke accessibility rule engine — axe-core's ruleset is used as-is; `ignoreRules` lets a team
  suppress specific rule IDs, not author new ones.
- Cross-browser accessibility/performance parity (Firefox/WebKit axe runs, mobile Lighthouse) —
  first version audits the same Chromium build Playwright already uses in CI.

## Architecture

No new package. Everything lives where the equivalent k6/ZAP checks already live, plus one small,
genuinely shared addition to the orchestrator.

### `@warden/runner` (extended)

Two new modules, each shaped exactly like `perf/k6.ts` / `security/zap.ts`: a pure result→CTRF
converter, a pure CTRF→gate evaluator, and one integration function (not unit-tested) that drives
the real tool.

```ts
// packages/runner/src/a11y/axe.ts

/** A single axe-core violation node (one offending DOM element). */
export interface AxeNode {
  target: string[]; // CSS selector path
  html: string;
  failureSummary?: string;
}

/** One axe-core rule violation, as returned by `axe.run()`. */
export interface AxeViolation {
  id: string; // rule id, e.g. 'color-contrast'
  impact: 'critical' | 'serious' | 'moderate' | 'minor' | null;
  description: string;
  help: string;
  helpUrl: string;
  tags: string[]; // e.g. ['wcag2aa', 'wcag143']
  nodes: AxeNode[];
}

/** The subset of `axe.run()`'s result Warden consumes, for one audited route. */
export interface AxeRouteResult {
  route: string;
  violations: AxeViolation[];
}

/** Injected impact→gate mapping, from `cfg.a11y`. */
export interface A11yGateConfig {
  blockOnImpact: AxeViolation['impact'][];
  warnOnImpact: AxeViolation['impact'][];
}

/**
 * Pure converter: one CTRF test per (route, violation). `extra` carries `impact`, `route`,
 * `helpUrl`, `wcagTags`, `selectors`, and `nodeCount` so a reviewer sees exactly what and where
 * without re-running the audit. Output is validated with `CTRFReportSchema`.
 */
export function axeResultsToCtrf(results: AxeRouteResult[]): CTRFReport;

/**
 * Pure gate mapping over the CTRF output of {@link axeResultsToCtrf}: any violation whose
 * `extra.impact` is in `blockOnImpact` → BLOCK, else any in `warnOnImpact` → WARN, else PASS.
 */
export function evaluateA11yGate(report: CTRFReport, cfg: A11yGateConfig): GateDecision;

export interface RunAxeOptions {
  standard: 'wcag2a' | 'wcag2aa' | 'wcag21aa' | 'wcag22aa';
  ignoreRules: string[];
  baseUrl: string;
  headless?: boolean;
}
export interface RunAxeResult {
  results: AxeRouteResult[];
  report: CTRFReport;
  gate: GateDecision;
}

/**
 * Integration glue. NOT unit-tested (launches a real Chromium via Playwright). For each route:
 * navigates, injects the bundled `axe-core` script, runs `axe.run({ runOnly: [standard] })`
 * filtered by `ignoreRules`, and closes the page. Converts + gates via the pure functions above.
 */
export function runAxeAudit(
  routes: string[],
  cfg: A11yGateConfig,
  opts: RunAxeOptions,
): Promise<RunAxeResult>;
```

```ts
// packages/runner/src/perf/lighthouse.ts

export type PerfMetric = 'performanceScore' | 'lcpMs' | 'tbtMs' | 'clsScore';

/** The subset of a Lighthouse JSON report Warden reads. */
export interface LighthouseReport {
  finalUrl?: string;
  categories?: { performance?: { score?: number } };
  audits?: Record<string, { numericValue?: number; score?: number }>;
}

/** One audited route's Lighthouse report, paired with the route it was resolved from. */
export interface LighthouseRouteResult {
  route: string;
  report: LighthouseReport;
}

/** Injected budgets, from `cfg.performance.browser.budgets`. */
export interface PerfBudgetConfig {
  performanceScoreMin: number; // 0..1
  lcpMs: number;
  tbtMs: number;
  clsScore: number;
  warnMarginPercent: number; // e.g. 10 → within 10% of budget still WARNs
}

/**
 * Pure converter: one CTRF test per (route, budgeted metric). `extra` carries `route`, `metric`,
 * `value`, and `budget`. A metric that breaches its budget is `failed`; a metric within
 * `warnMarginPercent` of its budget is `passed` but tagged `near-budget` so the gate can WARN.
 */
export function lighthouseResultsToCtrf(
  results: LighthouseRouteResult[],
  budgets: PerfBudgetConfig,
): CTRFReport;

/**
 * Pure gate mapping over the CTRF output of {@link lighthouseResultsToCtrf}: any `failed` metric
 * → BLOCK, any `near-budget` tag with no failures → WARN, else PASS.
 */
export function evaluatePerfBudgetGate(report: CTRFReport): GateDecision;

export interface RunLighthouseOptions {
  baseUrl: string;
  chromeFlags?: string[]; // defaults to ['--headless=new']
}
export interface RunLighthouseResult {
  results: LighthouseRouteResult[];
  report: CTRFReport;
  gate: GateDecision;
}

/**
 * Integration glue. NOT unit-tested (shells out to the `lighthouse` CLI per route, matching how
 * `runK6`/`runZapBaseline` shell out to their own binaries). Reads each route's JSON output,
 * converts + gates via the pure functions above.
 */
export function runLighthouseAudit(
  routes: string[],
  budgets: PerfBudgetConfig,
  opts: RunLighthouseOptions,
): Promise<RunLighthouseResult>;
```

```ts
// packages/runner/src/routes/resolve-changed-routes.ts
import type { ChangeSurface } from '@warden/core';

/** One entry of `cfg.a11y.routes` / `cfg.performance.browser.routes`. */
export interface RouteMapping {
  pathPrefix: string; // e.g. 'apps/checkout/app/'
  urlPattern: string; // e.g. '/checkout/*' — '*' is replaced with the path remainder
}

/**
 * Pure function: matches each changed file/module in `surface` against `mappings` by prefix,
 * substitutes the remainder into `urlPattern`, resolves against `baseUrl`, and dedupes. Returns
 * at most `maxRoutes` URLs (stable order: first-matched-first); anything beyond that is dropped,
 * not silently truncated — the caller reports how many were skipped.
 */
export function resolveChangedRoutes(
  surface: ChangeSurface,
  mappings: RouteMapping[],
  baseUrl: string,
  maxRoutes: number,
): { routes: string[]; skippedCount: number };
```

### `@warden/orchestrator` (extended)

One small, genuinely shared addition — folding several independently-computed `GateDecision`s
(functional tiers + a11y + perf budget) into the one decision the reporters already render:

```ts
// packages/orchestrator/src/combine-gate-decisions.ts
import type { GateDecision } from '@warden/core';

/**
 * Combines several independently-computed gate decisions into one, worst-first
 * (BLOCK > WARN > PASS). Reasons from every non-PASS decision are joined so the combined
 * decision explains every rule that fired, not just the first.
 */
export function combineGateDecisions(decisions: GateDecision[]): GateDecision;
```

Exported alongside the existing `evaluateExitCriteria` from `packages/orchestrator/src/index.ts`.
No changes to `ChangeSurface`, `TestTier`, or any existing orchestrator export — a11y and perf are
independent CTRF-producing checks that run alongside tiers, exactly like the existing k6/ZAP
checks, not a new `TestTier` value.

### `@warden/core` (no new types)

No new core types are needed. `GateDecision`, `CTRFReport`/`CTRFTest`, `Severity`, and
`ChangeSurface` already carry everything both checks need — matching the precedent of the existing
k6/ZAP modules, whose report shapes (`K6Summary`, `ZapReport`) live in `@warden/runner`, not core.
The only core change is the additive config below.

## Configuration

Two additive blocks. `a11y` is new; `performance` already exists (`enabled` / `p95LatencyMs`, for
the k6 API-latency check) and gains a `browser` sub-block for Lighthouse so API-latency and
browser-perf budgets stay clearly separate under one namespace.

```ts
a11y: {
  enabled: false,
  standard: 'wcag21aa',       // 'wcag2a' | 'wcag2aa' | 'wcag21aa' | 'wcag22aa'
  routes: [
    // { pathPrefix: 'apps/checkout/app/', urlPattern: '/checkout/*' },
  ],
  ignoreRules: [],             // axe rule ids to suppress, e.g. ['landmark-one-main']
  blockOnImpact: ['critical', 'serious'],
  warnOnImpact: ['moderate'],
  maxRoutesPerRun: 10,
},
performance: {
  enabled: false,
  p95LatencyMs: 500,           // existing: k6 API-latency budget
  browser: {
    enabled: false,
    routes: [
      // { pathPrefix: 'apps/checkout/app/', urlPattern: '/checkout/*' },
    ],
    budgets: {
      performanceScoreMin: 0.9,
      lcpMs: 2500,
      tbtMs: 300,
      clsScore: 0.1,
    },
    warnMarginPercent: 10,
    maxRoutesPerRun: 10,
  },
},
```

`routes` uses the same `{ pathPrefix, urlPattern }` shape for both blocks (a team with one route
map can reuse it for both checks). A repo with no `routes` configured runs neither check on any
route — a11y and perf are opt-in per repo and per route, never a full-site crawl.

## Data flow

1. A PR opens/updates; the CI job (Action or self-hosted runner) already computes the
   `ChangeSurface` via `@warden/orchestrator`'s `analyzeChangeSurface` for tier selection.
2. If `cfg.a11y.enabled` or `cfg.performance.browser.enabled`, the CLI's new `run-a11y` /
   `run-perf-budget` steps call `resolveChangedRoutes(surface, cfg.a11y.routes, baseUrl,
cfg.a11y.maxRoutesPerRun)` (and the perf equivalent) against a preview/staging deployment's
   `baseUrl`.
3. If no routes resolve, the step exits with a quiet neutral CTRF report (0 tests) and a PASS gate
   — "no changed routes matched `a11y.routes` / `performance.browser.routes`" — never a false BLOCK.
4. Otherwise, `runAxeAudit` launches Chromium via Playwright, visits each resolved route, injects
   axe-core, and collects violations; `runLighthouseAudit` shells out to the `lighthouse` CLI once
   per route and reads its JSON report.
5. Each produces a `CTRFReport` (`axeResultsToCtrf` / `lighthouseResultsToCtrf`) written to the
   same `warden-artifacts/` directory as every other tier's CTRF output (`a11y-report.json`,
   `perf-report.json`), and its own `GateDecision` (`evaluateA11yGate` / `evaluatePerfBudgetGate`).
6. `@warden/reporter`'s `aggregate()` merges every CTRF file in the directory — functional tiers,
   k6, ZAP, a11y, perf — into one report via the existing `mergeCtrf`, unchanged.
7. `combineGateDecisions([...])` folds the functional-tier gate (`evaluateExitCriteria`) together
   with the a11y and perf-budget gates into the one decision the merge gate publishes.
8. The existing reporters render it unchanged: the PR comment's coverage table gains rows for
   a11y/perf CTRF tests (tagged `wcag2aa`/`performance` etc.), the check-run gets annotations at
   each violation's route, and the dashboard's execution view shows them as regular `TestResult`s
   with `extra.helpUrl` / `extra.route` available to the UI.
9. A BLOCKing violation or budget breach blocks the merge exactly like a failed functional test;
   a WARN-level one (moderate impact, near-budget) surfaces without blocking.

## Safety & error handling

- **No routes resolved** → quiet neutral PASS with an explicit reason, not a false block; other
  checks still run. Matches the ZAP module's "nothing to scan" posture.
- **Capped, not silently truncated** — `maxRoutesPerRun` bounds both checks; `skippedCount` from
  `resolveChangedRoutes` is included in the step summary so a team knows work was left undone.
- **`lighthouse`/Chromium unavailable or the process errors** → the step fails loudly with a clear
  message (same posture as `runK6`/`runZapBaseline` today) rather than reporting a false PASS.
- **Known false positives** — `a11y.ignoreRules` suppresses specific axe rule IDs per repo, so a
  team isn't blocked by a rule that doesn't apply to their component library.
- **Lighthouse run-to-run variance** — a single run's LCP/TBT can be noisy; `warnMarginPercent`
  keeps near-budget results at WARN instead of a flaky BLOCK, and the summary always reports the
  raw measured values so a reviewer can judge for themselves (see Risks).
- **Idempotent** — re-running on the same head SHA overwrites `a11y-report.json` /
  `perf-report.json`, not appends, so the merge gate always reflects the latest run.
- **Preview-URL dependent** — both checks require a reachable `baseUrl` (a preview deployment);
  if the CI job has none, the step is skipped with a clear "no baseUrl configured" reason rather
  than silently auditing production or `localhost`.

## Testing

Fully hermetic for every pure unit, matching `perf/k6.test.ts` / `security/zap.test.ts`:

- `resolveChangedRoutes`: fixture `ChangeSurface` (via `fixtureChangeSurface` from
  `@warden/core/testing`) + route-mapping fixtures → asserted resolved URLs, dedup, and
  `skippedCount` when more routes match than `maxRoutes` allows.
- `axeResultsToCtrf`: fixture `AxeRouteResult[]` (critical/serious/moderate/minor across two
  routes) → asserted CTRF test count, `status`, and `extra.impact`/`extra.route`/`extra.helpUrl`.
- `evaluateA11yGate`: CTRF fixtures → BLOCK when a configured-blocking impact is present, WARN
  when only a warn-level impact is present, PASS on an empty violation set.
- `lighthouseResultsToCtrf`: fixture `LighthouseReport`s at/under/over each budget → asserted
  `failed` vs `passed`+`near-budget`-tagged CTRF tests with correct `extra.value`/`extra.budget`.
- `evaluatePerfBudgetGate`: CTRF fixtures → BLOCK on any breach, WARN on near-budget-only, PASS
  otherwise.
- `combineGateDecisions`: `[PASS, WARN, PASS]` → WARN with the WARN reason; `[WARN, BLOCK]` →
  BLOCK with both reasons joined; `[]` → PASS.
- `runAxeAudit` / `runLighthouseAudit`: NOT unit-tested, matching `runK6`/`runZapBaseline` — the
  pure pieces they delegate to are covered instead; exercised only in a dogfood CI run against a
  real preview deployment.
- `run-a11y.ts` / `run-perf-budget.ts` (CLI glue): unit-tested with an injected `surface`,
  injected `runAxeAudit`/`runLighthouseAudit` replaced by fakes returning canned `RunAxeResult` /
  `RunLighthouseResult`, and a temp `artifactsDir` — asserts the CTRF file is written and the
  returned gate matches the fake's decision. No real browser or network in these tests.

## Rollout

1. Ship `resolveChangedRoutes`, `axeResultsToCtrf`/`evaluateA11yGate`,
   `lighthouseResultsToCtrf`/`evaluatePerfBudgetGate`, and `combineGateDecisions` — fully
   hermetic, no CI wiring required yet.
2. Ship `runAxeAudit`/`runLighthouseAudit` and the `run-a11y`/`run-perf-budget` CLI commands;
   wire `warden.config`'s `a11y`/`performance.browser` blocks.
3. Extend `pr-report.ts` and the check-run reporter to render a11y/perf rows; extend the
   `warden-action` composite to run both steps when enabled, against the job's preview URL.
4. Dogfood against this repo's own dashboard app (a real Chromium-renderable frontend) before
   documenting `a11y`/`performance.browser` in `docs/configuration.md`.

## Risks & open items

- **Route-map maintenance burden** — `routes` mappings must be declared and kept current by hand;
  a repo with no mapping simply gets no coverage rather than a wrong one. A future `convention:
'file-based'` mode (Next.js `app/`/`pages/`-style auto-mapping) is a natural follow-on but is
  out of scope here to keep this a Low–Medium-effort change as scoped in §1.7.
- **Lighthouse variance** — single-run Lighthouse metrics are known to be noisy in shared CI
  runners; `warnMarginPercent` mitigates but doesn't eliminate flaky WARNs. A median-of-N-runs mode
  is a plausible follow-up once real-world variance is measured.
- **Preview-deployment dependency** — unlike functional tests (which can run against a local
  build), both checks need a reachable, representative `baseUrl`; repos without a preview-per-PR
  workflow will need one before either check is useful.
- **axe-core coverage ceiling** — axe-core (like every automated a11y tool) catches roughly a third
  to a half of WCAG success criteria; this check complements, not replaces, manual a11y review.
  The summary should say so explicitly rather than implying a passing gate means "accessible."
