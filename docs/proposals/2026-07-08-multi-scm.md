# Proposal: Multi-SCM Support (GitLab, Bitbucket, Azure DevOps)

- Status: Draft (design proposal) · Date: 2026-07-08 · Relates to: warden-next-competitive-gaps.md §2.4

## Summary

Warden's reporting, gating, and cross-repo coverage-sync surfaces are hard-wired to GitHub's REST
API (`Octokit`-shaped clients throughout `@warden/reporter`, `@warden/cli`, and
`@warden/coverage-sync`/`@warden/github-app`). This proposal introduces a `VcsProvider` seam in
`@warden/core` — `getPullRequest` / `getDiff` / `postComment` / `postStatus` / `openDraftPr` — and a
new `@warden/vcs` package with adapters for GitHub (already exists, reimplemented against the seam),
GitLab (merge requests), Bitbucket (pull requests), and Azure DevOps (pull requests). Every package
that talks to a host today is extended, additively, to accept an injected `VcsProvider` instead of a
GitHub-specific client, so the CI-embedded flow — run tests, post a comment, set a status, gate the
merge — works identically on all four hosts.

## Motivation

§2.4 of the competitive-gaps analysis is blunt about this: _"virtually every cloud QA tool integrates
all major hosts"_ — GitHub-only excludes a large share of enterprises, and Warden's cross-repo
coverage-sync App (`2026-07-08-cross-repo-coverage-sync.md`) is explicitly scoped GitHub-only today.
The tools named elsewhere in the gap analysis for enterprise reach (mabl, Testim, BrowserStack, Sauce
— §2.6) all ship first-class GitLab/Bitbucket/Azure DevOps integrations as table stakes, not an
add-on. An org standardized on GitLab or Azure Repos cannot adopt Warden's merge gate, PR comment, or
check-run reporting at all today — the gap isn't a missing feature, it's a missing _host_.

Warden already isolates its GitHub calls behind narrow, structurally-typed interfaces
(`OctokitIssuesClient`, `OctokitChecksClient` in `@warden/reporter`; `GitHubAccess` in
`@warden/core`'s coverage-sync types) specifically so they can be swapped for a fake in tests. That
same seam-first design makes generalizing the _host_, not just the test double, a natural extension
rather than a rewrite.

## Goals

1. Define one `VcsProvider` interface in `@warden/core` covering the operations every
   host-integrated surface actually needs: fetch a PR/MR + its diff, post a comment, post a
   commit/PR status, open (or idempotently update) a draft PR with file changes.
2. Ship adapters for GitHub, GitLab, Bitbucket, and Azure DevOps in a new `@warden/vcs` package,
   each built on an injected `fetch`, so no adapter ever makes a real network call in a unit test.
3. Make `@warden/reporter`'s PR-comment and check/status reporting `VcsProvider`-driven (additive
   new reporters), so `createReporters(cfg, deps)` works against any configured host.
4. Make `@warden/cli`'s `warden report` construct the right provider from `cfg.vcs.provider` and
   host-specific CI env vars, so the same CLI invocation works in GitHub Actions, GitLab CI,
   Bitbucket Pipelines, and Azure Pipelines.
5. Bridge the existing `GitHubAccess` seam (used by `@warden/coverage-sync` and
   `@warden/github-app`) onto any `VcsProvider`, so the coverage-sync _engine_ is host-agnostic even
   before host-specific webhook ingestion exists for it.
6. Zero behavior change for existing GitHub-only configs — `cfg.vcs.provider` defaults to `'github'`
   and the current Octokit-based reporters keep working unmodified.

## Non-Goals

- Webhook-hosted equivalents of `@warden/github-app` for GitLab/Bitbucket/Azure DevOps (a "GitLab
  App" / Bitbucket Connect app / Azure DevOps service hook receiver). Each host's app-hosting model
  differs enough to be its own proposal; this spec covers the CI-embedded flow (`warden report`
  inside the pipeline) for all four hosts and leaves cross-repo coverage-sync _hosting_ on non-GitHub
  hosts as follow-on work (see Rollout, Risks).
- Self-hosted/Server editions (GitHub Enterprise Server, GitLab self-managed, Bitbucket Server, Azure
  DevOps Server) beyond a configurable `baseUrl` — full API-version parity for Server editions is out
  of scope for V1.
- Rich, host-specific reporting features that have no cross-host equivalent (GitHub Check-Run file
  annotations at a specific line/range) — those stay on the existing `CheckRunReporter` as a
  GitHub-only capability; `VcsCheckReporter` intentionally targets the lowest common denominator.
- Any change to `@warden/orchestrator`'s change-surface computation — it already consumes
  `DiffFile[]` regardless of origin (per the cross-repo coverage-sync proposal), so it needs no
  changes here; this proposal only generalizes _how_ a diff gets fetched.

## Architecture

One new package, plus small additive extensions to three existing ones. The `VcsProvider` interface
itself lives in `@warden/core` — the same place `GitHubAccess` and `FileAccess` already live —
because every package that implements or consumes it needs the type without depending on the
adapter package.

### New: `@warden/core` (additive types — `packages/core/src/vcs.ts`)

```ts
/** The four supported hosts. Each has one VcsProvider adapter in @warden/vcs. */
export type VcsHost = 'github' | 'gitlab' | 'bitbucket' | 'azure-devops';

/**
 * A repository identifier generalized beyond GitHub's owner/repo. `project` is used only by
 * Azure DevOps, where a repo is scoped under `organization/project/repo`; `owner` carries the
 * org for Azure DevOps and GitLab, the workspace for Bitbucket, and the user/org for GitHub.
 */
export interface VcsRepoRef {
  host: VcsHost;
  owner: string;
  repo: string;
  project?: string; // Azure DevOps only
}

export interface VcsPullRequest {
  number: number; // PR / MR / pull request iid, whatever the host calls it
  title: string;
  url: string;
  headSha: string;
  headRef: string;
  baseSha: string;
  baseRef: string;
  author?: string;
  draft: boolean;
}

export type VcsCheckState = 'success' | 'neutral' | 'failure' | 'pending';

export interface VcsCheckStatus {
  /** A stable identifier for the check ('warden-qa'); reused across re-runs to update, not duplicate. */
  context: string;
  state: VcsCheckState;
  title: string;
  summary: string;
  detailsUrl?: string;
}

export interface VcsFileChange {
  path: string;
  content: string | null; // null = delete
}

export interface VcsDraftPrRequest {
  repo: VcsRepoRef;
  branch: string;
  baseBranch?: string; // defaults to the repo's default branch
  title: string;
  body: string;
  files: VcsFileChange[];
}

export interface VcsDraftPrResult {
  url: string;
  number: number;
}

/**
 * The one seam every SCM-touching Warden surface depends on — never a host SDK directly.
 * `@warden/vcs` ships one implementation per host; `@warden/core/testing` ships a fake.
 */
export interface VcsProvider {
  readonly host: VcsHost;
  getPullRequest(repo: VcsRepoRef, number: number): Promise<VcsPullRequest>;
  getDiff(repo: VcsRepoRef, number: number): Promise<DiffFile[]>;
  postComment(repo: VcsRepoRef, prNumber: number, body: string): Promise<void>;
  /** Creates or updates (by `context`) the status/check for a commit — Check Run, commit status, build status, or PR status, per host. */
  postStatus(repo: VcsRepoRef, headSha: string, status: VcsCheckStatus): Promise<void>;
  /** Idempotent: re-invoking for the same `repo`+`branch` updates the existing open PR/MR instead of opening a duplicate. */
  openDraftPr(request: VcsDraftPrRequest): Promise<VcsDraftPrResult>;
}
```

`DiffFile` is the existing `@warden/core` change-surface type (`packages/core/src/change-surface.ts`)
— unchanged, reused as-is, so `getDiff` output slots directly into
`computeChangeSurface(diff, cfg)` with no orchestrator changes.

### Extended: `@warden/coverage-sync` / `@warden/github-app` (no code changes — a new consumer)

`@warden/coverage-sync`'s `publish()` already depends only on the injected `GitHubAccess` interface
(`packages/core/src/coverage-sync.ts`), never on `@octokit/rest`. Because
`createGitHubAccessFromVcsProvider` (above) produces a value structurally satisfying that same
interface, `publish()` and the whole coverage-sync pipeline run against a GitLab, Bitbucket, or Azure
DevOps `VcsProvider` with **zero changes to `@warden/coverage-sync` or `@warden/github-app`**. What's
still missing for those hosts — and explicitly out of scope here — is a webhook _receiver_ to trigger
the pipeline on a merge-request/pull-request event outside GitHub (see Non-Goals, Rollout).

## Configuration

Additive `warden.config` block. Defaults preserve today's GitHub-only behavior exactly.

```ts
vcs: {
  // Which host this repo is hosted on. Drives which VcsProvider adapter is constructed.
  provider: 'github', // 'github' | 'gitlab' | 'bitbucket' | 'azure-devops'

  // Override for self-hosted/on-prem instances (GHES, GitLab self-managed, Bitbucket Server,
  // Azure DevOps Server). Defaults to each host's public API base URL.
  baseUrl: undefined as string | undefined,

  // Azure DevOps REST api-version pin (e.g. '7.1'). Ignored by other hosts.
  apiVersion: undefined as string | undefined,

  // Azure DevOps project name (owner/org comes from the repo path). Ignored by other hosts.
  project: undefined as string | undefined,
}
```

Tokens are **never** stored in `warden.config` — `createVcsProviderFromEnv` reads the host-specific
CI-provided secret (`GITHUB_TOKEN`, `GITLAB_TOKEN`, `BITBUCKET_TOKEN`, `AZURE_DEVOPS_TOKEN`), matching
how the GitHub Action already receives `GITHUB_TOKEN` today.

## Data flow

1. A CI job runs on a PR/MR — GitHub Actions, GitLab CI, Bitbucket Pipelines, or Azure Pipelines —
   and invokes `warden run` then `warden report` (the same two-step flow as today).
2. `warden report` reads `cfg.vcs.provider` (default `'github'`) and calls
   `createVcsProviderFromEnv(cfg, process.env)`, which resolves the `VcsRepoRef` and token from
   host-specific CI env vars and constructs the matching `@warden/vcs` adapter.
3. The runner (unchanged) has already produced a `TestExecution` (CTRF) from the local test run.
4. `createReporters(cfg, { vcs })` selects `CtrfReporter`, `GithubJobSummaryReporter` (GitHub Actions
   only — no-ops elsewhere per `cfg.reporting.githubJobSummary`), and — because `vcs` is present —
   `VcsCommentReporter` and `VcsCheckReporter` in place of the Octokit-only pair.
5. `VcsCommentReporter.report()` renders the same PR-report Markdown as today
   (`renderPrReport(execution, computeGateDecision(execution))`) and calls
   `provider.postComment(repo, ctx.prNumber, body)` — a PR comment on GitHub, an MR note on GitLab, a
   PR comment on Bitbucket, or a PR thread comment on Azure DevOps.
6. `VcsCheckReporter.report()` calls `provider.postStatus(repo, ctx.headSha, { context: 'warden-qa', state, title, summary })`
   with `state` derived identically from `computeGateDecision`, becoming a GitHub Check Run, a GitLab
   commit status, a Bitbucket build status, or an Azure DevOps PR status.
7. The merge gate's `PASS` / `WARN` / `BLOCK` decision — unchanged logic — is what each host's native
   merge-protection UI (required check / MR pipeline / PR build status / PR policy) surfaces to the
   reviewer; only the transport that carries it there is generalized.
8. For cross-repo coverage-sync: `createGitHubAccessFromVcsProvider(provider)` bridges the same
   `VcsProvider` into `GitHubAccess`, so a future non-GitHub webhook receiver could drive
   `@warden/coverage-sync`'s existing `publish()` unmodified — this proposal ships the bridge and
   proves it in tests; wiring a live receiver per host is follow-on work (Rollout phase 4/5).

## Safety & error handling

- **Auth failures** — each adapter wraps non-2xx responses in a `WardenError` with a host-prefixed
  code (`VCS_GITHUB_REQUEST_FAILED`, `VCS_GITLAB_REQUEST_FAILED`, `VCS_BITBUCKET_REQUEST_FAILED`,
  `VCS_AZURE_DEVOPS_REQUEST_FAILED`), including method/URL/status but never the token.
- **Idempotent draft PRs** — `openDraftPr` on every adapter first looks up an existing open PR/MR by
  source branch and updates it (matching `openOrUpdateDraftPr`'s existing GitHub behavior) rather
  than creating a duplicate on re-run.
- **Idempotent statuses** — `postStatus`'s `context` is reused across re-runs so a second CI attempt
  on the same commit _replaces_ the prior status instead of stacking a second one.
- **Bounded diffs** — `getDiff` paginates to a capped file count (matching the existing coverage-sync
  "bounded, nothing silently truncated" convention) and reports truncation in the returned
  `DiffFile[]` metadata rather than failing.
- **Suggestion-fidelity fallback** — `createGitHubAccessFromVcsProvider`'s `addPrSuggestions` renders
  native inline suggestions on GitHub/GitLab and a fenced-diff comment (via `postComment`) on
  Bitbucket/Azure DevOps, and the returned summary states which mode was used — never a silent
  degradation.
- **Least privilege** — documentation specifies the minimum token scope per host (`repo` +
  `pull_requests` equivalents only; no admin/webhook scopes needed for the CI-embedded flow).
- **Self-hosted `baseUrl`** — validated as an absolute URL at config-parse time via the existing
  `WardenConfigSchema`/`ConfigError` path; a malformed value fails fast at `loadConfig`, not mid-run.

## Testing

Fully hermetic, matching the rest of Warden — every adapter is tested against an injected `fetchImpl`
fake, never a live host:

- Each `*VcsProvider`: fixture request/response pairs per method (`getPullRequest`, `getDiff`,
  `postComment`, `postStatus`, `openDraftPr` both create-path and update-existing-path), asserting
  the exact request shape (URL, method, body) sent to the injected `fetch` and the mapped
  `VcsProvider`-shaped result — one test file per host, e.g. `gitlab-vcs-provider.test.ts` asserts a
  `getDiff` call hits `GET /projects/:id/merge_requests/:iid/diffs` and maps GitLab's diff shape to
  `DiffFile[]`.
- `create-vcs-provider.ts`: `cfg.vcs.provider` → the correct adapter class, with `baseUrl` /
  `apiVersion` / `project` threaded through.
- `github-access-bridge.ts`: given a `createFakeVcsProvider()`, asserts `openOrUpdateDraftPr` calls
  `openDraftPr`, `postCheckRun` calls `postStatus`, and `addPrSuggestions` calls `postComment` with
  the fenced-diff fallback body for a fake `host: 'bitbucket'` provider vs. inline-suggestion body for
  `host: 'github'`.
- `VcsCommentReporter` / `VcsCheckReporter`: `fakeVcsProvider` + a `TestExecution` fixture → asserts
  `postComment`/`postStatus` are called with the expected `repo`, `prNumber`/`headSha`, and a body
  matching `renderPrReport`'s existing snapshot.
- `create-reporters.ts`: asserts that passing `deps.vcs` selects `VcsCommentReporter`/`VcsCheckReporter`
  and that omitting it (existing callers) still selects `PrCommentReporter`/`CheckRunReporter`
  unchanged — a regression test pinning current GitHub-only behavior.
- `resolveVcsRepoRef` / `createVcsProviderFromEnv`: env-var fixtures per host (`GITHUB_REPOSITORY`,
  `CI_PROJECT_PATH`, `BITBUCKET_WORKSPACE`/`BITBUCKET_REPO_SLUG`, `SYSTEM_TEAMPROJECT`/
  `BUILD_REPOSITORY_NAME`) → the expected `VcsRepoRef`, and a missing-token case → `WardenError`.

## Rollout

1. `VcsProvider` + supporting types in `@warden/core`; `@warden/vcs` with `GitHubVcsProvider` (parity
   with today's `createFetchOctokit`) + `createFakeVcsProvider`; `Vcs*` reporters wired into
   `create-reporters.ts` behind the additive `deps.vcs`. No behavior change for existing configs.
2. `GitLabVcsProvider` — highest enterprise demand per §2.4 — wired into `cli/vcs-client.ts`;
   documented GitLab CI recipe (`.gitlab-ci.yml` snippet calling `warden run && warden report`).
3. `BitbucketVcsProvider` and `AzureDevOpsVcsProvider`, same pattern; documented Pipelines/Pipelines
   recipes.
4. `github-access-bridge.ts`; dogfood coverage-sync's `publish()` against a fake GitLab
   `VcsProvider` in an integration test to prove host-agnosticism end-to-end (still no live
   webhook receiver).
5. _(Future proposal)_ host-specific webhook receivers (a GitLab App equivalent, a Bitbucket Connect
   app, an Azure DevOps service-hook receiver) to bring cross-repo coverage-sync to full parity with
   the GitHub App on every host.

## Risks & open items

- **Suggestion-block parity** — Bitbucket and Azure DevOps have no native inline-suggestion API;
  the fenced-diff comment fallback is lower-fidelity (a reviewer applies it by hand) and should be
  clearly labeled in the coverage-sync summary when it's used.
- **Self-hosted/Server editions** — `baseUrl`/`apiVersion` cover the common case, but GHES, GitLab
  self-managed, Bitbucket Server, and Azure DevOps Server each have API differences (pagination,
  auth headers, endpoint paths) beyond a base-URL swap; full Server-edition parity is deferred.
- **Status/check granularity mismatch** — GitHub Check Runs support per-line file annotations;
  GitLab/Bitbucket/Azure statuses do not. `VcsCheckStatus` targets the common subset, so annotation
  detail remains a GitHub-only feature via the existing `CheckRunReporter` (see Non-Goals) — this
  should be called out clearly in docs so users don't expect line annotations on other hosts.
- **Token acquisition differs per host** — GitHub Actions auto-provisions `GITHUB_TOKEN`; GitLab CI,
  Bitbucket Pipelines, and Azure Pipelines each require the user to create and wire a PAT/repository
  variable manually. Docs need a per-host "getting the token" section, or first-run friction will be
  high.
- **No webhook receiver yet for non-GitHub hosts** — until Rollout phase 5, cross-repo coverage-sync
  is _usable_ (the engine is host-agnostic per this proposal) but not yet _triggered_ automatically
  on GitLab/Bitbucket/Azure DevOps; teams on those hosts get the CI-embedded gate/report flow now,
  the cross-repo App later.
