# Open questions — for your review

Decisions that need you. Ordered by impact. None block using the platform today.

## 1. Make the repository public? (the big one)
"Set up for open source eventually" — I did **not** flip it to public; that's your call and it exposes everything. Going public also **unlocks, for free**, two things you asked for that a private personal repo can't do:
- **Branch protection** (require PR + your approval before merge)
- **Secret scanning + push protection**

**Recommendation:** review the repo, then make it public when ready (`gh repo edit QuintinBotes/warden --visibility public`), and I (or you) can enable the rest immediately.

## 2. Branch protection (require your approval on every PR)
Configured intent: require a PR, **1 approving review**, **CODEOWNERS review** (you own everything → every PR needs your approval), required status checks (CI + Warden Self-Test), no force-push/deletion, **admin bypass ON** (so you're never locked out).
- On a **private personal** repo, GitHub may not enforce this (needs public or a paid plan). If my automated attempt failed, it's because of that — see the note in the final message.
- To enable once public: it's a single `gh api` call; I left the exact payload in the morning summary. CODEOWNERS + the PR template are already in place and will take effect the moment protection is on.

## 3. Disallowing forks — confirm the contribution model
You asked to disable forks. On a **private personal** repo this isn't a settable flag (and forks are already impossible without granted access). When you go public:
```bash
gh api -X PATCH repos/QuintinBotes/warden -F allow_forking=false
```
**Tradeoff to confirm:** with forks off, external contributors can't use the standard fork→PR flow — they'd need to be added as collaborators (or use "allow edits from maintainers" on a shared branch). That's maximally safe but higher-friction for drive-by contributors. Is that the model you want, or would you rather allow forks but rely on branch protection + required review to stay safe?

## 4. Add the `ANTHROPIC_API_KEY` secret
The self-test workflow's AI agent step is **gated off** until you add the secret:
`Settings → Secrets and variables → Actions → New repository secret → ANTHROPIC_API_KEY`.
Everything else (analyze, plan, the full test suite) runs without it.

## 5. Model & cost posture
Default model is `claude-sonnet-5`; the risk rules can bump high-risk PRs to Opus. Confirm the default and whether you want a stricter/cheaper posture. (The root `warden.config.ts` sets aggressive `highRiskPatterns` for dogfooding — soften if the self-test's risk scores look too high.)

## 6. Dashboard: static snapshot vs live SSR

This builds reliably and looks right. If you want the dashboard reading the live store on each request (SSR), that's a follow-up (needs `serverExternalPackages` for `better-sqlite3`). Want it?
